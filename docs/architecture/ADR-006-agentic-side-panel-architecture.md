# ADR-006: 도구 호출(Function Calling) 기반 에이전트 아키텍처

**상태**: Accepted  
**작성일**: 2026-03-31  
**관련 마일스톤**: M3 (v3 리팩토링)

---

## 1. Context
기존 GovOn v2는 사용자의 입력을 지정된 페르소나(분류기, 생성기 등)가 순차적으로 처리하는 **선형적 파이프라인** 구조였습니다. 그러나 v3에서는 공무원이 '민원 답변', '공문서 작성', '법령 검색' 등 다양한 업무를 비정형적으로 요청하므로, LLM이 상황에 맞게 기능을 선택하여 실행하는 **에이전트(Agentic)** 구조가 필요합니다.

## 2. Decision
단순 챗봇 구조를 파괴하고, **도구 호출(Tool-calling) 기반의 오케스트레이션 아키텍처**를 도입합니다.

### A. 주요 컴포넌트
1. **Agent Orchestrator (EXAONE v3)**: 사용자의 의도를 분석하고 필요한 도구를 결정하는 중앙 제어 장치.
2. **Tool Registry**: 사용 가능한 모든 행정 도구(API, 검색, 생성 모듈)의 명세와 실행 로직을 관리.
3. **Observation Context**: 도구 실행 결과(검색된 법령 등)를 실시간으로 모델의 컨텍스트에 주입하여 최종 답변의 근거로 활용.

### B. 기술 스택
- **Inference**: vLLM + EXAONE-Deep-7.8B (Tool-use 특화 튜닝)
- **Search**: FAISS (Dense) + BM25 (Sparse) 하이브리드 검색을 도구화
- **Orchestration**: ReAct (Reasoning and Acting) 패턴 적용

## 3. Consequences
- **장점**: 
  - 새로운 행정 업무(예: 예산 조회, 일정 관리) 추가 시 '도구'만 등록하면 되어 확장성이 뛰어남.
  - 사용자가 명확한 답변뿐만 아니라 "관련 법령만 찾아줘"와 같은 중간 단계 요청도 수행 가능.
- **단점**: 
  - 단순 생성 대비 추론 횟수가 늘어나 응답 속도(Latency)가 증가할 수 있음 (Streaming 최적화 필수).
  - 도구 선택 실패(Hallucination)에 대한 예외 처리가 복잡함.
