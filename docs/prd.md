## 1. 개요

### 1.1 프로젝트 비전
본 프로젝트는 **「대한민국 인공지능 행동계획(2026-2028)」**이 제시하는 '세계 1위 AI 정부' 도약의 핵심 가치인 **AI-Native 정부** 구현을 목표로 한다. 단순한 업무 보조 도구를 넘어, 공공 부문의 클라우드 네이티브 전환 과정에서 발생하는 보안 공백을 **온디바이스(On-device) 기술**로 해결하는 전략적 가교 역할을 수행한다.

### 1.2 핵심 솔루션: Edge-Cloud Hybrid AI
정부의 민간 클라우드 전환 정책을 준수하면서도 개인정보보호법(PIPA)을 완벽히 이행하기 위해, 보안이 핵심인 전처리는 **Edge(온프레미스)**에서, 고성능 추론 및 공공 데이터 활용은 **Cloud(민간/공공 클라우드)**에서 처리하는 하이브리드 아키텍처를 채택한다.

### 1.3 주 고객 및 비즈니스 영향
- **주 고객**: 민원 처리 및 분류를 담당하는 공직자 (실무자 및 관리자)
- **비즈니스 영향**:
    - **행정 효율**: AI 기반 자동 분류 및 초안 생성을 통한 민원 처리 사이클 단축.
    - **보안 완결성**: 클라우드 전송 전 온디바이스 PII 마스킹을 통한 법적 리스크 원천 차단.
    - **데이터 주권**: 핵심 행정 데이터 및 상담 이력의 로컬 관리로 국가 데이터 자산 보호.

## 2. 배경 및 전략적 타당성

### 2.1 정책적 배경: 대한민국 AI 행동계획(2026-2028)
2026년 2월 25일 국가인공지능전략위원회가 발표한 행동계획은 다음과 같은 GovOn의 추진 근거를 제공한다.
- **공공 AX 대전환 (p.85)**: 모든 행정 서비스를 데이터와 AI 기반으로 재설계하고, 모든 업무를 AI 학습 가능한 데이터로 기록.
- **범정부 AI 공통기반 구축 (p.92)**: 특정 모델 의존성을 탈피하고 언제든 최신 AI 모델로 교체 가능한 개방형 구조 지향.
- **클라우드 전환 의무화 (p.90)**: 기존 온프레미스 중심 체계를 민간 클라우드 중심으로 전면 전환.

### 2.2 핵심 문제: 클라우드 전환과 보안의 딜레마
정부의 클라우드 전환 기조에도 불구하고, 민원 시스템은 **개인정보 처리위탁 리스크**와 **국외 데이터 이전 문제**로 인해 전면적인 클라우드 AI 도입에 난항을 겪고 있다.
- **법적 제약**: 개인정보보호법 제17조 및 제18조에 따라 정보주체 동의 없는 외부 API 전송은 원칙적으로 금지됨.
- **보안 등급**: CSAP 상 등급 시스템 등 높은 보안 요구사항을 요구하는 행정망 데이터는 일반 클라우드 AI 서비스 활용이 불가능함.

### 2.3 솔루션: 엣지-클라우드 하이브리드 모델의 필요성
GovOn은 **보안(Edge)과 성능(Cloud)** 사이의 상충 관계를 기술적으로 해결한다.
1. **Edge (On-device/On-premise)**: 주민등록번호 등 민감정보 마스킹, 내부 보안 문서 기반 RAG 인덱스, OCR 파이프라인.
2. **Cloud**: 공개 법령·지침 기반 RAG, 대규모 모델(EXAONE-Deep 등)을 통한 복잡한 법리 해석 및 추론.
이 구조는 공무원이 안심하고 클라우드 AI의 성능을 활용할 수 있게 하는 **'보안 게이트웨이'**로서의 명분을 제공한다.

---

#### 문제점 2 (핵심) : 키워드 기반 분류의 **동음이의어·오분류** 한계

기존 시스템은 단순 키워드 매칭 방식으로 동작하므로 다음과 같은 오분류가 반복된다.

```
민원 예시: "도로 파손"  →  키워드: "파손"  →  [시설파손] 카테고리로 오분류
올바른 분류: [도로/교통] 카테고리
```

| 오분류 유형 | 예시 |
|------------|------|
| **동음이의어** | "배수" → 배수관 민원인데 산술 관련으로 분류 |
| **유사 표현** | "도로 파손" vs "보도 파손" 구분 불가 |
| **신조어·지역 방언** | 지역 특유 표현에 대응 불가 |
| **맥락 의존적 의미** | "소음 민원"이 공사·교통·주거 중 어느 카테고리인지 문맥 없이 판단 불가 |

오분류된 민원은 담당 부서가 바뀌어 처리 지연 및 이중 업무가 발생한다.

---

#### 문제점 3 (핵심) : 비효율적 수작업 프로세스

**담당 공무원 1인** 구조에서 민원 접수 → 분류 → 유사사례 검색 → 답변 작성의 전 과정을 수작업으로 처리해야 하며, 아래와 같은 비효율이 발생한다.

```
민원 1건당 처리 흐름 (현재):
민원 접수 → 읽기·내용 파악 → 분류 → 유사사례 수동 검색 → 답변 작성 → 검토·발송
```

- 국내 지자체 수: **243개** (광역 17개, 기초 226개) ([행정안전부 공식 문서](https://ko.wikipedia.org/wiki/%EB%8C%80%ED%95%9C%EB%AF%BC%EA%B5%AD%EC%9D%98_%EA%B8%B0%EC%B4%88%EC%9E%90%EC%B9%98%EB%8B%A8%EC%B2%B4_%EB%AA%A9%EB%A1%9D))
- 담당자의 경험·숙련도에 따라 답변 품질 편차가 크며, 신규 공무원의 경우 유사 사례를 찾는 데 더 많은 시간이 소요된다.

### 2.2 폐쇄망 환경에서 RAG가 필요한 이유

폐쇄망 환경에서 LLM 단독 운영 시 발생하는 치명적 한계가 있으며, 이를 RAG(Retrieval-Augmented Generation)로 보완한다.

#### LLM 단독 운영의 한계

```
[폐쇄망 LLM 단독 사용 시 문제]

LLM 학습 데이터 고정 (지식 컷오프 존재)
    │
    ├─ 최신 조례 개정 반영 불가
    ├─ 새로운 민원 유형 대응 어려움
    └─ 기관별 표준 답변 양식 업데이트 반영 불가

외부 검색엔진 연결 불가 (폐쇄망 특성상)
    │
    └─ 실시간 법령·지침 조회 불가능

모델 재학습 비용 부담
    │
    └─ 신규 데이터 반영 시마다 Fine-tuning 재수행 필요
       → GPU 자원·인력·시간 소모
```

#### RAG 도입으로 해결하는 문제

| 문제 | RAG 해결 방법 |
|------|--------------|
| **지식 고정 (Knowledge Cutoff)** | 최신 민원 사례·조례·지침을 벡터 DB에 주기적으로 추가 → 모델 재학습 없이 최신 정보 반영 |
| **기관별 맞춤 답변 부재** | 해당 기관의 실제 답변 이력을 벡터 DB로 구축 → 기관 특화 표현·서식 자동 참조 |
| **환각(Hallucination) 위험** | 검색된 실제 사례를 근거로 답변 생성 → 사실 기반 답변 신뢰성 향상 |
| **신규 공무원 적응** | 유사 사례 상위 5건 제시 → 경험 부족 공무원도 즉시 참조 가능 |

> **핵심 논거**: 폐쇄망에서는 외부 검색도 불가능하고 모델도 업데이트하기 어렵기 때문에, **내부 데이터 기반 RAG**가 실시간 데이터 활용의 유일한 현실적 대안이다.

## 3. 프로젝트 목표 및 KPI

### 3.1 핵심 목표
**OBJ-001: AI-Native 정부 기조에 부합하는 보안 완결형 민원 행정 지원**
- **보안 완결성**: 클라우드 AI 도입 시의 최대 걸림돌인 개인정보 유출 리스크를 온디바이스 전처리로 해결.
- **행정 지능화**: LLM 기반 자동 분류 및 답변 초안 생성을 통해 공무원의 단순 반복 업무 80% 이상 자동화.
- **정책 정합성**: 「대한민국 인공지능 행동계획」의 공공 AX 대전환 과제(p.85) 완벽 이행.

### 3.2 핵심 성과 지표 (KPI)
| KPI | 목표값 | 정책 및 기술적 근거 |
|-----|-------|-------------------|
| **PII 마스킹 정확도** | 99.9% | 개인정보보호법 준수를 위한 필수 요건 (온디바이스 Edge 처리) |
| **답변 생성 속도 (p95)** | < 3초 | 행정 효율 극대화를 위한 vLLM 및 가속화 기술 적용 |
| **BERTScore F1** | ≥ 85% | 사실 기반 답변 신뢰성 확보 (공공 AI 영향평가 가이드라인 반영) |
| **시스템 가용성** | ≥ 99.9% | 24/7 무중단 행정 서비스를 위한 엣지-클라우드 이중화 |

## 4. 사용자 및 페르소나 분석

> **주 고객**: 민원 처리 및 분류를 담당하는 공무원 (1인) vs **수혜자**: 신속한 답변을 받는 민원인 (다수)

### 4.1 페르소나 1: 민원 실무 담당자 (김민원, 35세)
- **Pain Points**: "민간 클라우드 AI가 좋다는데 보안 때문에 쓸 수가 없어요. 일일이 마스킹해서 올릴 수도 없고..."
- **Goal**: 보안 걱정 없이 최신 LLM 성능을 활용하여 민원 답변 초안을 1분 내에 생성하고 싶음.

### 4.2 페르소나 2: 정보보안 담당자 (박보안, 42세)
- **Pain Points**: "클라우드 전환은 국조실 방침이지만, 사고 나면 제가 책임져야 합니다. 데이터가 밖으로 나가는 건 절대 안 됩니다."
- **Goal**: 데이터가 물리적으로 외부망으로 나가기 전, 내부(Edge)에서 모든 민감정보가 제거됨을 보증받고 싶음.

## 5. 기술 아키텍처 (Hybrid Strategy)

### 5.1 하이브리드 역할 분담 (Edge-Cloud Split)
GovOn은 데이터의 보안 등급에 따라 처리 위치를 동적으로 결정하는 **Smart Orchestration** 기술을 적용한다.

| 구분 | Edge (On-premise Gateway) | Cloud (Public/Private AI Cloud) |
|------|---------------------------|---------------------------------|
| **처리 대상** | 민감 민원 원문, 개인식별정보(주민번호 등) | 비식별화된 요약본, 공개 법령 데이터 |
| **핵심 기능** | PII 마스킹, 내부 지침 RAG, OCR | EXAONE-Deep 추론, 글로벌 지식 검색 |
| **보안 등급** | **등급 '상'** (기밀/개인정보) | **등급 '중/하'** (공개 가능 정보) |

### 5.2 시스템 구성도 (Logical View)
```
[사용자 환경] <───> [GovOn Edge Node] <─── (Secure Tunnel) ───> [GovOn Cloud Hub]
      │                     │                                       │
      │ 1. 민원 접수        │ 2. 온디바이스 PII 마스킹              │ 4. 고성능 LLM 추론
      │ 6. 최종 답변 확정   │ 3. 로컬 RAG (내부 이력)               │ 5. 범용 답변 초안 생성
      └─────────────────────┴───────────────────────────────────────┘
```

### 5.3 상세 기술 스택
- **Inference Server**: vLLM (PagedAttention 최적화)
- **Quantization**: AWQ 4-bit (VRAM 사용량 66% 절감, RTX 3060급 구동 가능)
- **Connectivity**: Model Context Protocol (MCP) - 범정부 AI 공통기반 연동 표준
- **Vector DB**: FAISS (Hybrid Search: Vector + BM25)
- **Preprocessing**: Ko-ELECTRA (PII 개체명 인식) + 정규표현식 하이브리드 마스킹

### 5.4 데이터 파이프라인 (Security-First)
1. **Ingestion**: 행정망 내 민원 접수.
2. **De-identification**: 클라우드 전송 전, Edge 노드에서 소형 LLM을 활용해 모든 개인정보 마스킹 처리.
3. **Transmission**: 비식별화된 '벡터' 또는 '마스킹 텍스트'만 클라우드로 전송.
4. **Augmentation**: 클라우드 상의 최신 법령/조례 데이터를 참조하여 컨텍스트 보강.
5. **Synthesis**: 최종 답변 초안을 생성하여 Edge로 반환.


**현재 상태**:

## 6. 범위 정의 및 로드맵

### 6.1 프로젝트 범위 (In-Scope)
- **Edge Module**: 온디바이스 PII 마스킹 엔진, 로컬 RAG 인덱서, Edge Gateway API.
- **Cloud Module**: EXAONE-Deep 기반 고성능 추론 엔진, 공개 법령 DB 통합 검색.
- **Hybrid Controller**: 보안 등급에 따른 데이터 라우팅 및 세션 관리 로직.
- **Agent Interface**: 하이브리드 추론 결과를 통합 스트리밍으로 보여주는 대화형 UI.

### 6.2 로드맵
- **M1 (기획/설계)**: 하이브리드 아키텍처 확정 및 법적 타당성 검토 완료.
- **M2 (MVP 구현)**: 온디바이스 마스킹 및 클라우드 연동 프로토타입 개발.
- **M3 (고도화/최적화)**: AWQ 양자화 적용 및 MCP 표준 규격 연동.
- **M4 (검증/마무리)**: 공공 AI 영향평가 자가 진단 및 최종 보안 감사 대응.

## 7. 보안 및 윤리 가이드라인 (공공 AI 영향평가 대응)

행동계획 p.97의 **'공공 AI 영향평가'** 및 p.98의 **'공공부문 AI 윤리기준'**을 준수한다.

### 7.1 데이터 주권 및 개인정보 보호
- **Zero-PII Outflow**: 모든 개인 식별 정보는 Edge망을 벗어나기 전 반드시 마스킹 처리됨을 보장.
- **Local Sovereignty**: 상담 이력 등 핵심 행정 자산은 기관 내부 서버에만 저장하며, 클라우드에는 학습용/로그용으로 전송하지 않음.

### 7.2 알고리즘 투명성 및 설명 가능성 (XAI)
- **Source Citation**: RAG를 통해 답변 생성 시, 근거가 된 법령 조항 및 유사 사례 출처를 명확히 제시.
- **Bias Mitigation**: AI Hub 데이터를 활용한 지속적인 편향성 테스트(행동계획 p.54) 실시 및 필터링 레이어 구축.

### 7.3 시스템 안정성 (Resilience)
- **Offline Fallback**: 클라우드망 장애 시, Edge의 경량 모델만으로 최소한의 분류 및 마스킹 기능이 동작하도록 설계.
